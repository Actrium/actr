# **Actor 食谱：构建强大服务的实用模式**

欢迎来到 actr-rtc 的厨房！本文档是《开发者指南》的进阶篇，它将超越基础 API 的讲解，专注于展示如何利用“宏观 Actor”这一核心架构，构建出健壮、可扩展、可维护的分布式服务。

这里的每一个“模式”，都不是在弥补框架的“限制”，而是在**发挥框架的优势**。我们将向您展示，如何将“一个进程一个 Actor”的设计，转化为一种战略性的、简化复杂系统设计的强大武器。

---

## **模式一：“服务门面”模式 (Service Façade)**

#### **问题场景**

当应用逻辑变得复杂，需要处理用户、会话、计费等多个不同领域时，如何在一个 Actor 进程内，保持代码的组织性、可读性和高内聚？

#### **核心思想**

将宏观 Actor 视为一个清晰、稳定的**服务入口 (Façade)**。它的职责是接收外部请求，然后将其干净地委托给内部的、按领域划分的、高内聚的业务模块。Actor 本身不实现具体业务逻辑，只做顶层协调。

#### **实现步骤**

1.  **定义内部模块**: 为每个业务领域创建独立的 `struct`。
    ```rust
    // a_module.rs
    pub struct A_Module {
        // ...模块自身的状态
    }
    impl A_Module {
        pub async fn do_something(&self) { /* ... */ }
    }

    // b_module.rs
    pub struct B_Module {
        // ...
    }
    impl B_Module {
        pub async fn do_another_thing(&self) { /* ... */ }
    }
    ```

2.  **创建 Façade Actor**: 主 Actor 持有所有模块的实例。
    ```rust
    // application_actor.rs
    struct ApplicationActor {
        module_a: A_Module,
        module_b: B_Module,
    }
    ```

3.  **委托实现 `trait`**: 在为 `Actor` 实现由 `.proto` 生成的 `trait` 时，方法体只做委托，保持极度简洁。
    ```rust
    #[async_trait]
    impl IApplicationService for ApplicationActor {
        // 这个 rpc 调用属于 A 领域
        async fn some_a_action(&self, req: A_Request, _ctx: Arc<Context>) -> Result<A_Response, Status> {
            // 直接委托给 A 模块
            self.module_a.do_something(req).await
        }

        // 这个 rpc 调用属于 B 领域
        async fn some_b_action(&self, req: B_Request, _ctx: Arc<Context>) -> Result<B_Response, Status> {
            // 直接委托给 B 模块
            self.module_b.do_another_thing(req).await
        }
    }
    ```
**优势**: 这种模式让你的 Actor 内部结构像一个清晰的微服务架构，每个模块职责单一，易于独立开发和测试，完美地解决了在单一进程内管理复杂性的问题。

## **模式二：“外部资源”模式 (External Resources)**

#### **问题场景**

我的 Actor 需要访问外部资源，如数据库连接池、Redis 客户端或配置管理器。如何安全、高效地管理这些资源的生命周期和共享访问？

#### **核心思想**

通过**依赖注入 (Dependency Injection)**，在应用的“根部”（即 `main.rs`）一次性地创建所有共享资源，然后通过 `Actor` 的构造函数，将其所有权 (`Arc`) 注入 `Actor` 内部。坚决避免使用全局静态变量。

#### **实现步骤**

1.  **在 `main.rs` 中创建资源**:
    ```rust
    // main.rs
    #[tokio::main]
    async fn main() {
        // 1. 在应用启动时，创建所有需要的共享资源
        let db_pool = Arc::new(DatabasePool::connect("...").await.unwrap());
        let redis_client = Arc::new(RedisClient::new("...").unwrap());

        // 2. 通过构造函数注入
        let actor = ApplicationActor::new(db_pool, redis_client);

        // 3. 启动系统
        ActorSystem::new(actor_id).attach(actor).start().await?;
    }
    ```

2.  **在 `Actor` 中持有并使用资源**:
    ```rust
    // application_actor.rs
    struct ApplicationActor {
        db: Arc<DatabasePool>,
        redis: Arc<RedisClient>,
    }

    impl ApplicationActor {
        // 构造函数接收依赖
        pub fn new(db: Arc<DatabasePool>, redis: Arc<RedisClient>) -> Self {
            Self { db, redis }
        }
    }

    #[async_trait]
    impl IApplicationService for ApplicationActor {
        async fn get_user(&self, req: GetUserRequest, _ctx: Arc<Context>) -> Result<User, Status> {
            // 在业务逻辑中直接、安全地使用注入的资源
            let user = self.db.get_user_by_id(req.user_id).await?;
            Ok(user)
        }
    }
    ```
**优势**: 依赖关系清晰，易于管理和替换。在测试中，你可以轻易地注入一个“模拟”的数据库连接，而无需一个真实的数据库，极大地提升了单元测试的速度和稳定性。

## **模式三：“并发句柄”模式 (Concurrent Handle)**

#### **问题场景**

我的服务需要在并发的“快车道”上处理高吞吐量的数据，同时又需要安全地与“状态路径”中的核心状态交互。如何实现？

#### **核心思想**

Actor 为每一个快车道任务，创建一个专属的、线程安全的**并发句柄 (Concurrent Handle)**。快车道回调函数捕获的是这个轻量级的句柄，而不是 Actor 自身。句柄将所有需要修改核心状态的操作，都封装成向 Actor 的“状态路径”发送消息 (`tell`) 的异步方法，从而保证了线程安全。

#### **实现步骤**

```rust
// 在 Actor 的某个 State Path 方法中 (例如 OpenStream)
async fn open_stream(&self, ctx: Arc<Context>) {
    // 1. 创建一个代表此流的并发句柄
    let handle = self.create_stream_handle(ctx);

    // 2. 创建快车道回调，它捕获的是句柄
    let callback = move |chunk| {
        let handle = handle.clone();
        tokio::spawn(async move {
            // 3. 在回调中调用句柄的方法，语法简洁
            handle.on_chunk(chunk).await;
        });
    };

    // 4. 注册回调
    ctx.register_fastpath_callback(stream_id, callback);
}

// --- 并发句柄的实现 ---
struct StreamHandle {
    actor_id: ActorId,
    context: Arc<Context>,
    // ... 其他只读或线程安全的状态
}

impl StreamHandle {
    // 这个方法需要修改 Actor 核心状态，所以它内部必须是消息发送
    pub async fn on_chunk(&self, chunk: Bytes) {
        let result = process(chunk); // 可在当前线程执行的纯计算
        let msg = Message::ChunkProcessed(result);
        // 将状态变更的“意图”发回给 State Path
        self.context.tell(self.actor_id, msg).await;
    }
}
```
**优势**: 此模式完美地隔离了并发环境和串行环境。它既为开发者提供了简洁的 API (`handle.on_chunk`)，又将线程安全的复杂性完全封装在了句柄内部，是实现高性能、高安全性的关键。

## **模式四：“主题发布/订阅”模式 (Topic-based Pub/Sub)**

#### **问题场景**

如何实现一个通用的发布/订阅系统，让客户端可以订阅任意“主题”（如 `stock.price.goog`），并接收发布到该主题的所有消息？

#### **核心思想**

使用一个 Actor 作为消息总线。其核心状态就是一个 `Topic -> Subscribers` 的映射表。`Subscribe` 和 `Publish` 请求都经由“状态路径”串行化处理，以保证订阅列表的线程安全。

#### **实现步骤**

1.  **定义契约 (`pubsub.proto`)**:
    ```protobuf
    service PubSubService {
      rpc Subscribe(SubscribeRequest) returns (Empty);
      rpc Publish(PublishRequest) returns (Empty);
    }
    message SubscribeRequest { string topic = 1; }
    message PublishRequest { string topic = 1; bytes data = 2; }
    message Publication { string topic = 1; bytes data = 2; }
    ```

2.  **实现 `PubSubActor`**:
    ```rust
    #[derive(Default)]
    struct PubSubActor {
        // 核心状态: Topic -> Set<ActorId>
        subscriptions: Mutex<HashMap<String, HashSet<ActorId>>>,
    }

    #[async_trait]
    impl IPubSubService for PubSubActor {
        async fn subscribe(&self, req: SubscribeRequest, ctx: Arc<Context>) -> Result<Empty, Status> {
            let subscriber_id = ctx.get_caller_id();
            let mut subs = self.subscriptions.lock().await;
            subs.entry(req.topic).or_default().insert(subscriber_id);
            Ok(Empty {})
        }

        async fn publish(&self, req: PublishRequest, ctx: Arc<Context>) -> Result<Empty, Status> {
            let subs = self.subscriptions.lock().await;
            if let Some(subscribers) = subs.get(&req.topic) {
                let publication = Publication { topic: req.topic, data: req.data };
                for sub_id in subscribers {
                    // 使用 tell 将消息广播给所有订阅者
                    ctx.tell(sub_id.clone(), publication.clone()).await?;
                }
            }
            Ok(Empty {})
        }
    }
    ```
**优势**: 这是一个构建实时通知系统、事件总线或任何需要“一对多”消息分发场景的通用基础模块。

## **模式五：“分布式会话”模式 (Distributed Session)**

#### **问题场景**

如何管理一个有明确生命周期、有成员列表的、多用户参与的“会话”（如游戏对局、聊天会话）？

#### **核心思想**

这是“服务门面”和“发布/订阅”模式的组合应用。使用一个 Actor 作为“会话”状态的“真相来源”，管理其生命周期和成员列表。其内部的消息广播，则是 Pub/Sub 模式的一个具体应用。

#### **实现步骤**

1.  **定义契约 (`session.proto`)**: 定义加入、离开、发送消息等特定于会话的动作。
    ```protobuf
    service ChatSessionService {
      rpc Join(JoinRequest) returns (Empty);
      rpc Leave(LeaveRequest) returns (Empty);
      rpc SendMessage(ChatMessage) returns (Empty);
    }
    ```

2.  **实现 `ChatSessionActor`**:
    ```rust
    #[derive(Default)]
    struct ChatSessionActor {
        // 核心状态: SessionID -> { members: HashSet<ActorId>, ...other_state }
        sessions: Mutex<HashMap<String, SessionState>>,
    }

    #[async_trait]
    impl IChatSessionService for ChatSessionActor {
        async fn join(&self, req: JoinRequest, ctx: Arc<Context>) -> Result<Empty, Status> {
            let mut sessions = self.sessions.lock().await;
            let session = sessions.entry(req.session_id).or_default();
            session.members.insert(ctx.get_caller_id());
            // ... 此处可以广播一个“用户加入”的通知
            Ok(Empty {})
        }

        async fn send_message(&self, req: ChatMessage, ctx: Arc<Context>) -> Result<Empty, Status> {
            let sessions = self.sessions.lock().await;
            if let Some(session) = sessions.get(&req.session_id) {
                let broadcast_msg = BroadcastMessage { ... };
                // 应用 Pub/Sub 模式：遍历成员并 tell
                for member_id in &session.members {
                    if member_id != &ctx.get_caller_id() {
                        ctx.tell(member_id.clone(), broadcast_msg.clone()).await?;
                    }
                }
            }
            Ok(Empty {})
        }
    }
    ```
**优势**: 这个模式展示了如何将通用的基础模式（如 Pub/Sub）组合起来，构建出具有丰富领域逻辑的、有状态的、高层级的应用程序。
