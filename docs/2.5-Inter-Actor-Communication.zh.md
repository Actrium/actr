# **Actor 间通信：`LocalActor` 与 `RemoteActor`**

在 actr-rtc 框架中，任何通信都涉及到“视角”问题。为了清晰地描述通信的双方，我们定义了两个核心概念：`LocalActor` 和 `RemoteActor`。

理解这两个概念，是掌握框架通信模型的关键。

---

### **1. `LocalActor` (本地 Actor)**

`LocalActor` 指的是**在当前进程中**运行的、由你亲手实现其业务逻辑的那个 Actor 实例。

*   **角色**: 服务的**提供方**。
*   **代码体现**: 它是在 `main.rs` 中被实例化并 `.attach()` 到 `ActorSystem` 的那个 `struct`。在它自己的方法内部，它就是 `self`。

```rust
// 在 main.rs 中

// MyEchoActor 就是一个 LocalActor 的具体实现
#[derive(Default)]
struct MyEchoActor;

#[async_trait]
impl IEchoService for MyEchoActor { ... }

// 实现 AttachableActor trait
impl AttachableActor for MyEchoActor {
    type Adapter = EchoServiceAdapter;
}

// 实例化我们的 LocalActor
let local_actor = MyEchoActor::default();

// 将 LocalActor 附加到系统  
let system = ActorSystem::new(actor_id).attach(local_actor);
```

`LocalActor` 是你所有业务逻辑和状态的“家”。

---

### **2. `RemoteActor` (远程 Actor)**

`RemoteActor` 指的是一个**本地的、类型安全的代理对象**，它代表了网络中另一个进程里运行的 `Peer` (对等端)。

*   **角色**: 服务的**调用方/消费者**。
*   **代码体现**: 它通常是通过 `ActorSystem` 的方法（如 `connect`）返回的句柄。你持有这个句柄，并调用其上的方法来间接地与远程的 `LocalActor` 交互。

```rust
// 假设 system 已经启动

// 1. 获取一个指向远程 Peer 的 RemoteActor 句柄
//    这里的 IEchoService 是由 proto 生成的 trait
let remote_actor: RemoteActor<dyn IEchoService> = system.connect("peer-id-of-echo-service").await?;

// 2. 通过句柄，调用远程服务
let response = remote_actor.request(EchoRequest { message: "hello" }).await?;
```

`RemoteActor` 是你与外部世界沟通的“电话”。它封装了所有网络通信的复杂性，让你感觉就像在调用一个本地方法一样。

---

### **3. 总结：对称的通信模型**

`LocalActor` 和 `RemoteActor` 的概念共同构成了一个对称的通信模型：

*   你的应用程序的核心是实现一个 `LocalActor` 的业务逻辑。
*   当你的 `LocalActor` 需要调用另一个服务时，它会获取一个代表该服务的 `RemoteActor` 句柄。

这个模型完美地适配了各种网络拓扑：

*   **在 P2P 场景中**: 两个对等端都运行着自己的 `LocalActor`，并可以互相持有对方的 `RemoteActor` 句柄，实现双向通信。
*   **在 C/S 场景中**: 服务器上运行着你的 `LocalActor`。一个原生的桌面应用或 Web 应用（通过专用的 SDK）则会获取一个 `RemoteActor` 句柄来调用你的服务。

通过清晰地划分这两个概念，框架得以在不同场景下都提供一致、清晰且类型安全的编程体验。
