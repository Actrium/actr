# ActorSystem 与 Actor

我们可以用一个比喻来理解：
> **`ActorSystem` 是舞台、是灯光、是音响、是剧院本身。而 `Actor` 是站在舞台上表演的演员。**

演员不需要知道灯光是如何供电的，他只需要一个“追光灯”的指令。剧院则为演员提供表演所需的一切环境和支持。

### **本框架的 Actor 模型**

为了帮助来自不同技术背景的开发者快速理解，这里明确阐述本框架 Actor 模型的核心特点与保证。

#### **与经典 Actor 模型的对比**

本框架并没有遵循原教旨主义的 Actor 模型，而是只保留了最核心的“状态隔离”和“消息驱动”两个概念。简单说有 `Send` 和 `Receive`，没 `Spawn`。对于 Actor 收到消息之后的处理方式，开发者可以自由选择，我们称之其为“**宏观 Actor**”（在 `1-Concepts—and-Architecture.zh.md` 中已经阐述了这一理念）。

*   **相同之处**:
    *   **状态隔离**: Actor 的内部状态由其自身管理，外部无法直接访问，只能通过消息来交互。
    *   **消息驱动**: Actor 的行为由接收到的消息驱动。执行逻辑是异步的。
    *   **位置透明**: (通过 `ActorId` 和 `Context`) 调用者无需知道目标 Actor 的物理位置。

*   **不同之处**:
    *   **粒度**: 本框架的 Actor 是“宏观”的，通常代表一个完整的业务服务单元，而不是像 Akka 中那样代表一个微小的计算任务。一个应用中通常只有少数几组 Actor。
    *   **监督机制**: 本框架**不提供**内建的、类似 Erlang/Akka 的监督树和自动重启策略。Actor 的错误处理和恢复逻辑需要由开发者在业务代码中自行实现（例如，通过 `try/catch` 结合 `Context` 的能力）。Actor 的生命周期与服务进程或连接会话绑定。
    *   **Mailbox 的可见性**: 虽然内部使用 Mailbox（邮箱）来调度状态变更消息，但开发者通常不直接与其交互，而是通过类型安全的 `service` 契约和 `Context` 的 `call/tell` 方法来通信。

#### **并发模型与状态管理**

这是理解本框架最核心、也最容易产生误解的部分。得准确理解。

框架的并发模型基于一个核心原则：**框架保证对消息的 *处理入口* 是串行的，但开发者需要自行保证 *处理过程* 的线程安全。**

**1. 框架的保证：串行化的消息处理入口**

对于所有经由 **状态路径 (State Path)** 的消息，框架的 `Scheduler` 保证在任意时刻，**只会调用一个消息处理逻辑**。换句话说，框架不会并发地进入两个不同的 `rpc` 方法或 `tell` 消息的处理器。这个机制是 `Actor` 模型“状态隔离”思想的基础，它为您提供了一个安全地变更状态的“时机”。

**2. 开发者的责任：保证处理过程的线程安全**

重要的是，框架的上述保证**仅限于“入口”**。一旦代码进入了您的 `Actor` 方法实现，您就进入了标准的 Rust 多线程异步环境。如果您的代码在方法内部创建了新的并发任务，那么这些新任务将**不再受**框架的串行化入口保护。

*   **危险的例子**:
    ```rust
    // 在一个 Actor 的方法中
    async fn handle_my_message(&self, msg: MyMessage) {
        // 在这里直接修改 self.state 是安全的，因为框架保证了没有其他
        // handle_* 方法在同时运行。
        self.state.counter += 1;

        // **危险操作!**
        // 下面的 tokio::spawn 会将闭包交由 tokio 运行时并发执行。
        // 如果这个闭包和另一个消息的处理器都尝试修改 self.state，
        // 就会立刻产生数据竞争！
        let state_clone = Arc::clone(&self.state);
        tokio::spawn(async move {
            // **错误**: 这里产生了并发访问！
            state_clone.lock().unwrap().counter += 1; 
        });
    }
    ```

*   **正确做法**:
    *   **默认原则**: 坚持在 `Actor` 的 `async` 方法内部，**不使用 `tokio::spawn` 或 `std::thread::spawn`**。所有逻辑都在当前被调度的任务中完成。对于 I/O 密集型操作，直接在方法体中 `.await` 即可。
    *   **共享状态**: 如果您确实需要在 `Actor` 内部使用共享的可变状态（例如，在 `Fast Path` 回调和 `State Path` 方法之间共享），您**必须**使用标准的同步原语来保护它，例如 `tokio::sync::Mutex` 或 `std::sync::RwLock`。
    *   **CPU 密集型任务**: 如果遇到必须并发执行的 CPU 密集型任务，正确的做法是将其发送到**一个独立的、由您自行管理的计算线程池**，并通过 `tokio` 的消息通道将计算结果发回给 `Actor` 处理。

**总结**: 框架为您提供了一个串行的“事件分发循环”，但这层保证仅限于处理消息的入口。它不限制您的业务代码在内部创建新的并发任务，这些新任务的线程安全需要您自行负责。将 `Actor` 的状态安全地封装在 `async` 方法的边界内，是使用本框架的最佳实践。

#### **`ActorSystem` 的职责 (环境与支撑)**

`ActorSystem` 是 Actor 运行的**宿主环境**，是一个重量级对象，它的职责包括：

1.  **生命周期管理 (Lifecycle Management)**
    *   负责整个应用的**启动**和**关闭**。
    *   实现**优雅关停**逻辑（监听 `SIGTERM`，清空邮箱等）。
    *   `.start()` 方法是它开始工作的入口。

2.  **运行时环境 (Runtime Environment)**
    *   拥有并管理底层的 `tokio` **异步运行时**和**线程池**。
    *   Actor 的代码只是在 `ActorSystem` 提供的线程上被调度执行。

3.  **I/O 与网络 (I/O & Networking)**
    *   管理底层的 WebRTC 连接和数据通道。
    *   拥有（我们之前提过的）**`Handler` (输入处理器)**，负责所有流量的初始分发。

4.  **核心服务提供 (Core Services)**
    *   管理**定时器**后台（用于 `schedule_tell`）。
    *   提供**日志**基础设施。
    *   拥有**快车道注册表 (`Fast Path Registry`)** 等核心数据结构。

5.  **`Context` 的创建与注入 (Context Provider)**
    *   `ActorSystem` 的一项核心职责，是在调度每一个状态路径上的消息时，为其构建并注入一个 `Context` 对象，其中封装了该消息的来源、追踪 ID 等环境信息。
    *   这个 `Context` 对象是 `Actor` 与 `ActorSystem` 交互的唯一、安全的**桥梁**。它将 `ActorSystem` 提供的强大能力（如日志、定时器）安全地暴露给 `Actor` 的业务逻辑。

#### **`Actor` 的职责 (业务逻辑与状态)**

`Actor` 是**业务逻辑**和**状态**的承载者。它是一个相对轻量级的对象，专注于“做什么”而不是“怎么做”。

1.  **业务逻辑实现 (Business Logic Implementation)**
    *   `Actor` 负责实现由 `.proto` 文件生成的 `trait`。
    *   它的代码完全是关于“当收到 `X` 消息时，根据我当前的状态，应该执行 `Y` 操作，并可能改变我的状态到 `Z`”。

2.  **使用 `Context`**
    *   `Actor` **不是**服务的提供者，而是服务的**消费者**。
    *   它不管理定时器，而是通过 `context.schedule_tell` **请求**一个定时消息。
    *   它不管理网络连接，而是通过 `Context` **请求** `ActorSystem` 将消息发送给其他对等端（例如，通过 `ctx.tell` 或 `ctx.publish`）。

---

### **结论与应用**

有了这个清晰的划分：

*   **API 语义明确**: `ActorSystem::new().attach(...)` 变得理所当然，因为 Actor 的逻辑需要“附加”到作为“剧院”的 `ActorSystem` 上才能运行。
*   **代码可测试**: 我们可以独立地测试 `Actor` 的业务逻辑，只需模拟 `Context` 对象即可，完全不需要启动一个 `ActorSystem` 或任何网络组件。
*   **关注点分离**: 框架开发者可以专注于优化 `ActorSystem` 的性能和稳定性。应用开发者则可以专注于 `Actor` 内部的业务逻辑实现。
