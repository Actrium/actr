# 项目清单 (actr.toml) 与命令行工具 (actr-cli)

actr-rtc 框架提供了一套可选的自动化工具链，包含一个项目清单文件 `actr.toml` 和一个命令行工具 `actr-cli`。其设计目标是将项目构建、运行、依赖管理等操作进行封装，以提供一个统一、便捷的工作入口，让开发者可以更多地关注业务逻辑。

---

## 1. 项目清单文件 (`actr.toml`)

`actr.toml` 是 actr-rtc 框架项目的核心配置文件。它的主要作用是为自动化构建工具（如 `actr-cli`）提供一个统一的信息来源。

该文件主要包含三个部分：

1.  **`[package]`**: 项目的基础元信息。
2.  **`[dependencies.protos]`**: 项目所依赖的外部 Protobuf 契约。
3.  **`[routing]`**: 系统内的消息路由规则。

### **1.1. `[package]` - 项目元信息**

这部分遵循了 `Cargo.toml` 的标准约定，用于描述当前 Actor 项目的基本信息。

```toml
[package]
name = "my-actor-service"
version = "0.1.0"
edition = "2021"
```

### **1.2. `[dependencies.protos]` - 外部契约依赖**

为了实现服务间的解耦，框架推荐将依赖的外部服务接口通过此部分进行声明。构建工具 (`actr-cli`) 会读取此列表，在编译前自动获取这些远程文件。

```toml
[dependencies.protos]
# key "user_service" 是一个本地别名，用于方便管理
# value 是一个包含了依赖来源和版本的表

# 示例一：从 Git 仓库获取
user_service = { git = "https://github.com/my-org/user-protos.git", path="user/v1/user.proto", tag = "v1.2.0" }

# 示例二：从一个固定的 URL 获取
email_service = { url = "http://schema.my-company.com/email.v1.proto" }

# 示例三：依赖本地文件（主要用于 monorepo 项目）
analytics_service = { path = "../common/protos/analytics.proto" }
```

*   **`git`**: Git 仓库的 URL。
*   **`path`**: （用于 Git 依赖）指明 `.proto` 文件在仓库中的相对路径。
*   **`tag` / `branch` / `rev`**: （用于 Git 依赖）指定获取的具体 Git 标签、分支或提交哈希。
*   **`url`**: 一个可以直接下载 `.proto` 文件的 HTTP/HTTPS URL。

### **1.3. `[routing]` - 消息路由规则**

这部分定义了系统级的、基于消息类型的路由表。它的设计目的是将消息的“发送”与“处理”解耦，使得服务的拓扑结构更加灵活。

```toml
[routing]
# key 是消息的完整类型名 (package.MessageName)，用引号包裹
# value 定义了该消息的目标和交互类型

# 规则一：一个 Call 类型的路由
"user.v1.GetUserRequest" = { call = "user.v1.UserService" }

# 规则二：一个 Tell 类型的路由
"email.v1.SendEmailRequest" = { tell = "email.v1.EmailService" }

# 规则三：一个 Pub/Sub 类型的路由
"analytics.v1.UserRegisteredEvent" = { publish = "user-registration-events" }
```

*   **`call`**: 用于**请求-响应**式交互。当一个 Actor 调用 `ctx.request(GetUserRequest{...})` 时，框架会根据此规则，查找实现了 `user.v1.UserService` 的 Actor 进程，并将请求路由给它。
*   **`tell`**: 用于**单向**消息。当一个 Actor 调用 `ctx.notify(SendEmailRequest{...})` 时，框架会根据此规则，查找实现了 `email.v1.EmailService` 的 Actor 进程，并将消息路由给它。
*   **`publish`**: 用于**发布-订阅**模式。消息将被发送到一个逻辑“主题”，所有订阅了此主题的 Actor 都会收到它。（注意：订阅关系的管理由具体的 Pub/Sub Actor 自身负责，这里只定义了发布时的路由）。

通过将路由规则从业务代码中剥离到配置文件中，开发者可以在不修改 Actor 代码的情况下，调整分布式系统中服务间的拓扑关系和数据流向。

## 2. 命令行工具参考 (`actr-cli`)

`actr-cli` 是与 `actr.toml` 配套的命令行工具。

### **2.1. `actr-cli build`**

此命令用于执行项目的完整构建流程。

**用法**
```bash
actr-cli build
```

**执行流程**

运行 `actr-cli build` 时，该工具会按顺序执行以下步骤：

1.  **读取清单**: 解析当前目录下的 `actr.toml` 文件。
2.  **拉取依赖**: 检查 `[dependencies.protos]` 部分，通过 Git 或 HTTP 下载所有声明的远程 `.proto` 依赖，并根据声明对版本进行管理。
3.  **准备环境**: 创建一个临时的编译目录，并将项目自身的 `proto` 文件和所有下载好的依赖 `proto` 文件准备好。
4.  **生成代码**: 检查项目是否存在 `src/main.rs`。如果不存在（即采用“自动运行器模式”），则根据 `actr.toml` 配置自动生成一个标准入口文件。随后，调用 `protoc` 和框架的编译器插件，并配置好所有导入路径 (`--proto_path`)，为所有涉及的 `.proto` 文件生成业务 `trait` 和 `Adapter`。
5.  **编译项目**: 调用语言对应的构建工具（如 `cargo build`），编译业务逻辑代码和所有生成的代码，最终产生可执行文件。

### **2.2. `actr-cli run`**

此命令用于运行一个已编译的 Actor。

**用法**
```bash
# 运行 debug 构建产物
actr-cli run

# 运行 release 构建产物
actr-cli run --release

# 传递参数给 Actor 进程
actr-cli run -- --my-arg value
```

**执行流程**

1.  该命令会检查项目是否已构建，如果需要，会先执行构建步骤。
2.  然后，它会执行 `target/{debug|release}/<package_name>` 可执行文件，并将 `--` 之后的所有参数透传给该进程。

### **2.3. `actr-cli init`**

此命令用于快速创建一个新的 actr-rtc 项目骨架。

**用法**
```bash
actr-cli init my-new-actor
```

**执行流程**

该命令会创建一个名为 `my-new-actor` 的新目录，并填充一个最小化的项目模板，包括：
*   `Cargo.toml`
*   预先配置好的 `actr.toml`
*   `protos/greeter.proto` 示例文件
*   `src/main.rs` 示例实现

### **2.4. 其他规划中的命令**

*   `actr-cli test`: 运行项目测试。
*   `actr-cli add proto <url>`: 向 `actr.toml` 中添加一个新的远程 proto 依赖。
*   `actr-cli publish`: 将项目的 proto 契约发布到 Schema Registry。
