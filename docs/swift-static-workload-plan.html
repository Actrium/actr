Swift 开发 static workload 方案（ACTR）

目标：让 Swift workload 能走 Node::link(...)（静态链接）路径，而不是 .attach(package)。
结论：新增 “Rust trait ⇄ C ABI ⇄ Swift 实现” 三层桥接，并补齐代码生成与构建集成。

1. 架构设计

1.1 新增桥接抽象（Rust）

  • 在 core/hyper 新增 FfiLinkedWorkloadHandle（或 SwiftWorkloadAdapter）。
  • 该 adapter 实现现有 linked workload 所需接口，内部通过函数指针调用 Swift。
  • 保持 Node::link 不变，对外只新增 Node::link_ffi(...) 或内部自动分发。

1.2 C ABI 回调表

  • 定义 ActrWorkloadVTable：包含 on_start/on_ready/on_stop/on_error/dispatch 等函数指针。
  • 统一内存协议：请求/响应字节缓冲区由 Rust 分配与释放，Swift 只读写 payload。
  • 统一错误协议：返回 ActrErrorCode + message bytes，映射到 Rust ActorResult。

1.3 Swift 侧 runtime

  • 新增 Swift runtime 包：注册 workload 实例并导出 C 符号（@_cdecl）。
  • Swift dispatcher 基于 route_key 调用具体 handler。
  • protobuf 编解码由生成代码处理，业务只实现 handler。

2. 模块改造清单

模块改造内容产出
core/framework 抽离最小可 FFI 化生命周期与 dispatch 契约 稳定 ABI 结构体 + 错误码定义
core/hyper 新增 FFI linked adapter；把 envelope/ctx 转给回调 link_ffi 路径可跑通
bindings/ffi 声明 C ABI、buffer 生命周期 API、错误映射 API actr_ffi.h / Rust extern 接口
protoc-gen + actr gen 生成 Swift dispatcher + handler 协议 + route map Swift 可直接实现业务 workload
cli / examples 新增 Swift static workload 示例与构建脚本 端到端 demo + 文档

3. 分阶段实施（建议 4 个 PR）

PR-1：ABI 基础设施

  • 定义 ActrWorkloadVTable、buffer API、error API。
  • 加 Rust 单测：ABI roundtrip、错误码映射、内存释放一致性。

PR-2：Hyper 接入

  • 实现 FfiLinkedWorkloadHandle，打通 dispatch 主链路。
  • 加集成测试：mock FFI workload 能完成 request/response。

PR-3：Swift 代码生成与 runtime

  • 生成 Swift handler 协议、dispatcher、注册入口。
  • 实现 Swift runtime shim（导出 C 符号 + workload 实例管理）。

PR-4：示例与文档

  • 新增 examples/swift/static-echo。
  • 补充开发文档（Swift 工具链、构建、运行、调试）。

4. 验收标准

  • Rust workspace cargo fmt --all、cargo check 通过。
  • Swift static example 可由 host 通过 link_ffi 成功启动并处理 RPC。
  • 生命周期钩子（start/ready/stop/error）在 Swift workload 可观测。
  • 无内存泄漏（buffer 分配释放闭环可验证）。

5. 关键风险与规避

  • ABI 漂移：给 ActrWorkloadVTable 增加 abi_version，启动时强校验。
  • 内存边界错误：严格单向所有权，统一 alloc/free API，不允许双方随意释放。
  • 并发模型差异：明确 callback 线程模型，必要时在 Rust 侧串行化 dispatch。
  • 生成代码升级成本：把 Swift 生成模板版本化，和 proto schema 绑定。

6. 最小 MVP 范围

  • 先只支持 unary RPC（不做 streaming）。
  • 先支持 on_start/on_stop/dispatch 三个核心回调。
  • 先覆盖一个 echo 服务，跑通端到端后再补齐其余事件回调。
