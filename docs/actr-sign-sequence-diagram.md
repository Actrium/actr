# Actr Sign & Auth Full Pipeline Sequence Diagrams

> **Core signing logic: developers sign packages with their private key and publish to the platform, which hosts the public key; users download the package and verify the signature with the public key — confirming the package was published by that developer and has not been tampered with. (Note: currently the key pair is generated by the platform — needs review)**

## Known Issues

### 1. AIS registration should verify Manufacturer, not package publish records

Currently `verify_actr_type()` checks the `mfr_package` table for a matching `type_str` record. This causes Native Mode actors with non-reserved names to always fail (since they never go through `actr pkg publish`). In practice, the Hyper layer already performs full package integrity verification (Ed25519 signature + SHA-256 hash), so AIS does not need to re-verify package contents — it only needs to confirm that the **Manufacturer exists and is in Active status**.

**Solution**: Change `verify_actr_type()` from `lookup_package(type_str)` to querying the `mfr` table (Manufacturer table), verifying only that the `actr_type.manufacturer` record exists with `status == Active`. Reserved names (acme/self/actrix) continue to bypass directly. The `mfr_package` table is retained for package distribution and version management, but no longer serves as a prerequisite for registration authorization.

**Current implementation** ([issuer.rs](file:///Users/zhj/RustProject/Actrium/actrix/crates/services/ais/src/issuer.rs#L604-L637)):

```rust
async fn verify_actr_type(&self, actr_type: &ActrType) -> Result<(), AidError> {
    let type_str = format!("{}:{}:{}", actr_type.manufacturer, actr_type.name, actr_type.version);

    if actrix_mfr::reserved::is_reserved(&actr_type.manufacturer) {
        return Ok(());  // reserved names bypass
    }

    // ❌ Queries mfr_package table — Native Mode never publishes, always fails
    let valid = actrix_mfr::manager::lookup_package(&pool, &type_str).await?;
    if !valid {
        return Err(AidError::ManufacturerNotVerified);
    }
    Ok(())
}
```

**Proposed change**:

```rust
async fn verify_actr_type(&self, actr_type: &ActrType) -> Result<(), AidError> {
    if actrix_mfr::reserved::is_reserved(&actr_type.manufacturer) {
        return Ok(());  // reserved names bypass
    }

    // ✅ Query mfr table — only check Manufacturer exists and is Active
    let mfr = Manufacturer::get_by_name(&pool, &actr_type.manufacturer)
        .await?
        .ok_or(AidError::ManufacturerNotVerified)?;

    if mfr.status != MfrStatus::Active {
        return Err(AidError::ManufacturerNotVerified);
    }
    Ok(())
}
```

### 2. PSK renewal not implemented

AIS always returns `psk: None` when issuing credentials. Every registration goes through the full flow with no lightweight renewal.

**Solution**: Generate an HMAC-SHA256 PSK in `issue_credential` (using `actr_id + actr_type + realm_id + expires_at` as input) and return it with `RegisterOk`. Hyper client uses the PSK to call `/ais/renew` before credential expiry.

### 3. Package distribution logic missing

`actr pkg publish` only registers metadata (manifest text + signature) to MFR — it does not upload the `.actr` file. MFR cannot distribute packages to Hyper nodes, nor can it verify that binary_hash actually matches the binary content. Currently MFR's `publish_package` only verifies the manifest signature (`crypto::verify_signature`) but not the package contents (actrix does not depend on the `actr_pack` crate).

**Solution — upload the full package**:

1. **Change `publish` to upload the entire `.actr` package**. Since `build` already packages the signature (`actr.sig`) into the `.actr` file, the publish request no longer needs a separate `signature` field:
   ```bash
   # Before: actr pkg publish --package .actr --keychain mfr.json --endpoint URL
   # After:  actr pkg publish --package .actr --endpoint URL
   ```
   After MFR receives the `.actr` package:
   - Extract `actr.toml` from ZIP → get manufacturer / name / version
   - Extract `actr.sig` from ZIP → get signature
   - Look up the MFR public key for that manufacturer in the database
   - Call `actr_pack::verify(&package_bytes, &mfr_pubkey)` for full verification (signature + binary hash)
   - On success → store `.actr` package in object storage (S3/MinIO) → record in `mfr_package` table

2. **Add `actr pkg pull <actr_type>` command** to download `.actr` packages from MFR:
   - MFR adds `GET /mfr/pkg/download/{type_str}` endpoint
   - Hyper nodes download and verify with `actr_pack::verify()`

### 4. MFR server generates the developer's private key (design question)

During MFR registration, the Ed25519 key pair is generated server-side and the private key is returned via HTTP response ([manager.rs L109](file:///Users/zhj/RustProject/Actrium/actrix/crates/services/mfr/src/manager.rs#L109)). Actrix does not store or use the private key (returned once only), but industry standard (Apple/npm/SSH) is for the developer to generate the key pair locally and only upload the public key. Is this intentional or a design concern?

**Solution**: Switch to developer-local key generation via `actr pkg keygen`. Upload only the public key during `/mfr/apply` — the server never generates or touches the private key.
