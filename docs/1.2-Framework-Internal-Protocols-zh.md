# 1.2 框架内部协议 (Framework Internal Protocols)

> [!NOTE]
> 本文档定义了 **框架自身** 使用的、标准化的底层 Protobuf 协议。这些是构成框架通信能力的基石。
>
> 如果您想了解如何为 **您的应用业务** 定义 `service` 契约，请参阅《[3.3 应用 Protobuf 契约](./3.3-Application-Proto-Contract.zh.md)》。

本篇文档是框架所有**内置 Protobuf 契约**的唯一、权威的参考源。

---

## 1. WebRTC 基础契约 (`webrtc.proto`)

此文件定义了对 WebRTC 标准协商消息的通用封装，作为一个独立的、可重用的基础协议层。它不包含任何与上层 Actor 或信令逻辑相关的字段。

### 1.1 `webrtc.proto` 完整定义

```protobuf
syntax = "proto3";

package webrtc;

// IceCandidate 对应 WebRTC 标准中的 RTCIceCandidateInit 接口。
message IceCandidate {
  string candidate = 1;
  optional string sdp_mid = 2;
  optional uint32 sdp_mline_index = 3;
  optional string username_fragment = 4;
}

// SessionDescription 是对 SDP 的统一封装。所有轨道信息都包含在 sdp 字符串中。
message SessionDescription {
  enum Type {
    OFFER = 0;
    ANSWER = 1;
    RENEGOTIATION_OFFER = 2;
  }
  Type type = 1;
  string sdp = 2;
}
```

## 2. Actor 核心契约 (`actor.proto`)

此文件定义了框架中关于“Actor”自身的核心概念，包括其身份（Identity）和访问控制规则（ACL）。它是所有上层业务逻辑的基础。

### 2.1 `actor.proto` 完整定义

```protobuf
syntax = "proto3";

package actor;

// Actor 的类型码，用于区分不同种类的 Actor
enum ActorTypeCode {
  PREDEFINED     = 0; // 预定义的系统级 Actor
  ANONYMOUS      = 1; // 代表未认证的匿名用户
  AUTHENTICATED  = 2; // 代表已认证的用户或服务
}

// ActorType 描述了一个 Actor 的类别
message ActorType {
  ActorTypeCode code = 1;
  optional string manufacturer = 2; // 可选的厂商或组织前缀，用于命名空间隔离
  string name = 3;                // 在类别下的具体名称
}

// ActorId 是一个 Actor 在分布式系统中的唯一、结构化身份
message ActorId {
  uint64 serial_number  = 1;
  ActorType type        = 2;
}

// AclRule 定义了一条访问控制规则，它基于 ActorType
message AclRule {
  ActorType source_type = 1;
  ActorType target_type = 2;
  enum Permission {
    DENY = 0;
    ALLOW = 1;
  }
  Permission permission = 3;
}

// Acl 是一组访问控制规则的集合
message Acl {
  repeated AclRule rules = 1;
}
```

## 3. 信令层契约 (`signaling.proto`)

此文件定义了“身份感知”的信令路由协议。它负责理解 Actor 的身份，并根据 ACL（访问控制）规则，在 Actor 之间可靠地转发消息。

### 3.1 `signaling.proto` 完整定义

```protobuf
syntax = "proto3";
import "google/protobuf/timestamp.proto";
import "webrtc.proto"; // 引入 WebRTC 基础类型
import "actor.proto";  // 引入 Actor 核心定义

package signaling;

// SignalingMessage 是信令服务器与客户端之间所有通信的顶层信封。
message SignalingMessage {
  oneof message_type {
    WebRTCSignal webrtc_signal = 1;
    NewActor new_actor = 2;         // <--- NewPeer -> NewActor
    ActorList actor_list = 3;       // <--- PeerList -> ActorList
    PullAcl pull_acl = 4;
    actor.Acl acl = 5;              // <--- 引用 actor.Acl
    SignalingError error = 9;
  }
}

// WebRTCSignal 封装了一次具体的点对点 WebRTC 信令交互。
// 它直接包含了路由信息和具体的协商载荷。
message WebRTCSignal {
  actor.ActorId source_actor_id = 1; // <--- 全面使用 ActorId
  actor.ActorId target_actor_id = 2;
  oneof payload {
    webrtc.IceCandidate ice_candidate = 3;
    webrtc.SessionDescription session_description = 4;
  }
}

// NewActor 用于通知网络中出现了一个新的、且你有权访问的 Actor。
message NewActor {
  actor.ActorId actor_id = 1;
}

// ActorList 是 ActorId 列表的封装。
message ActorList {
  repeated actor.ActorId actor_ids = 1;
}

// PullAcl 用于请求获取当前的 ACL 规则集。
message PullAcl {}

// SignalingError 封装了详细的错误信息。
message SignalingError {
  uint32 code = 1;
  string message = 2;
  optional bytes request_id = 3;
}
```

### 3.2 核心设计解析

*   **身份感知的路由**: 信令层协议直接使用结构化的 `actor.ActorId` 进行路由。这使得信令服务器能够理解每个通信端点的身份类型，是实现 ACL 的基础。
*   **发现即授权**: `NewActor` 消息的设计，隐含了信令服务器的核心职责：在广播 `NewActor` 前，必须先根据 `actor.Acl` 规则进行过滤，确保对等端只会收到它有权发现和连接的 Actor 的信息。
*   **清晰的分层**: 整个协议栈被清晰地划分为三层：`webrtc.proto` (传输内容) -> `actor.proto` (身份与规则) -> `signaling.proto` (路由与控制)，职责明确，高度内聚。

## 4. Actor 生命周期契约 (原生接口/Trait)

与网络通信的信令层不同，Actor 的生命周期钩子（如 `on_start`）是作为一个**原生语言接口**（例如 Rust 的 `trait`，Go 的 `interface`）来定义的，**而不是**一个 Protobuf 契约。

这样做是基于以下设计考量：
*   **性能**: 生命周期事件是 `ActorSystem` 与其管理的 Actor 之间的本地调用。使用原生接口可以避免不必要的序列化开销。
*   **开发者体验**: 开发者可以用自己最熟悉的、最符合语言习惯的方式来实现这些核心方法。

### 4.1 `ILifecycle` Trait 定义 (以 Rust 为例)

```rust
#[async_trait]
pub trait ILifecycle {
    /// Called after the actor is attached and the system is started.
    async fn on_start(&self, ctx: Arc<Context>);

    /// Called before the actor is stopped.
    async fn on_stop(&self, ctx: Arc<Context>);
}
```
