Swift Static Workload 实现方案

目标是在 Swift/iOS/macOS 应用内直接实现 ACTR workload 业务逻辑，并通过现有 Rust runtime 的 Node::link() 路径运行，不再要求业务 workload 打包成独立 .actr package。

结论

推荐方案是复用现有 UniFFI callback bridge，把 Swift workload 包装为 Rust 侧 DynamicWorkload，再由新的 FFI 构造器调用 Node::link()。 这不是让 Swift 直接实现 Rust trait，也不是让 Swift 产出独立 .a 供 ACTR 动态识别。

最小改动 Recommended 现有 WorkloadLifecycleBridge、DynamicWorkload 已经能承载 Swift 回调。
运行模型 Swift app、UniFFI 生成层、Rust runtime 在同一进程内运行，workload 不经过包签名和二进制加载。
边界 package 模式仍保留；新增 linked/static Swift 入口，不改变 .actr package 行为。

现状依据

位置 当前能力 缺口
core/hyper/src/lib.rs Node::link<W: actr_framework::Workload>() 已是 static link 入口。 入口只在 Rust API 层公开，Swift 不能直接调用 Rust 泛型 trait。
bindings/ffi/src/workload.rs WorkloadLifecycleBridge 和 DynamicWorkload 已把外语回调转换成 Rust workload。 没有 FFI runtime 构造器把 DynamicWorkload 送进 Node::link()。
bindings/ffi/src/runtime.rs ActrNode::new_from_package_file() 已封装 package-backed 流程。 当前只支持 attach(package)，没有 linked/static 构造器。
bindings/swift SwiftPM 通过 UniFFI 生成 bindings 和 XCFramework 分发。 需要新增 Swift-friendly 的 high-level API 和文档示例。

目标 API

Swift 调用形态

final class EchoWorkload: WorkloadLifecycleBridge {
    func onStart(ctx: ContextBridge) async throws {}
    func onReady(ctx: ContextBridge) async throws {}
    func onStop(ctx: ContextBridge) async throws {}
    func onError(ctx: ContextBridge, event: ErrorEventBridge) async throws {}

    func dispatch(ctx: ContextBridge, envelope: RpcEnvelopeBridge) async throws -> [UInt8] {
        switch envelope.routeKey {
        case EchoRequest.routeKey:
            let request = try EchoRequest(serializedBytes: envelope.payload)
            let response = EchoResponse.with { $0.message = request.message }
            return Array(try response.serializedData())
        default:
            throw ActrError.internal(msg: "unknown route: \(envelope.routeKey)")
        }
    }
}

let workload = DynamicWorkload(
    lifecycle: EchoWorkload(),
    signaling: nil,
    websocket: nil,
    webrtc: nil,
    credential: nil,
    mailbox: nil
)

let node = try await ActrNode.fromLinkedWorkload(
    configPath: "/path/to/actr.toml",
    workload: workload
)
let actor = try await node.start()

Rust FFI 构造器

#[uniffi::constructor(async_runtime = "tokio")]
pub async fn new_from_linked_workload(
    config_path: String,
    workload: Arc<DynamicWorkload>,
) -> ActrResult<Arc<Self>> {
    let init = Node::from_config_file(&config_path).await?;
    crate::logger::init_observability(init.runtime_config().observability.clone());

    let attached = init.link(LinkedDynamicWorkload::new(workload)).await?;
    let ais_endpoint = attached.ais_endpoint().to_string();
    let registered = attached.register(&ais_endpoint).await?;

    Ok(Arc::new(Self {
        inner: Mutex::new(Some(registered)),
        network_event_handle: Mutex::new(None),
    }))
}

架构流

Swift app

实现 WorkloadLifecycleBridge，负责 protobuf 解码、业务处理和 response 编码。

→
UniFFI layer

DynamicWorkload 持有 Swift callback，对 Rust 暴露 actr_framework::Workload。

→
Rust runtime

Node::link() 连接 workload，注册 AIS，启动节点并处理 inbound RPC。

inbound RPC 的关键路径是：network message → LinkedWorkloadHandle::dispatch → DynamicDispatcher → Swift dispatch → protobuf response bytes。

分层实现步骤

  1. Rust FFI runtime 入口

    在 bindings/ffi/src/runtime.rs 引入 DynamicWorkload， 新增 ActrNode::new_from_linked_workload(config_path, workload)。 行为与 new_from_package_file 对齐，但把 attach(package) 替换为 link(workload)。

  2. DynamicWorkload wrapper

    如果 Arc<DynamicWorkload> 不能直接满足 Node::link() 的类型边界， 新增 LinkedDynamicWorkload newtype，在 Rust 侧实现 actr_framework::Workload，内部转发到 DynamicWorkload。

  3. Swift high-level API

    在 bindings/swift/Sources/Actr 增加 Swift-friendly wrapper， 暴露 ActrNode.fromLinkedWorkload(configPath:workload:)，避免用户直接接触低层生成命名。

  4. Service metadata 策略

    第一阶段允许 service_spec = None，用 actr.toml 的 actor metadata 注册。 第二阶段再增加可选 serviceSpec 参数或从 Swift proto descriptor 输入生成 metadata。

  5. 示例与文档

    新增一个 Swift echo workload 示例，展示 route dispatch、protobuf decode/encode、启动和停止流程。 更新 bindings/swift/README.md 与 bindings/swift/docs/api.md。

关键设计决策

使用 UniFFI callback，而不是 Swift 静态库 ABI。

Swift 直接输出一个 ACTR 可识别的 static library 需要定义 C ABI symbol、vtable、async 回调、内存所有权和 protobuf route 协议。 仓库里已有 DynamicWorkload 和 callback interfaces，复用它能把工作量集中在 runtime 构造器和 Swift API 上。

service_spec 第一阶段不强行生成。

linked/static 模式没有 package manifest，无法像 package 模式一样自动从 .actr 包提取 proto exports。 第一阶段先注册 actor type；需要服务发现暴露 schema 时，再增加显式 service spec 输入。

不要把 Node<S> typestate 直接暴露给 Swift。

Swift README 已声明 typestate 是 Rust power-user API。Swift 侧应继续提供一站式构造器，保持移动端 SDK 的简单表面。

行为断言

  • Swift onStart、onReady、dispatch 至少各触发一次。
  • unknown routeKey 返回明确错误，不静默掉包。
  • package-backed new_from_package_file() 行为不变。
  • linked/static 模式无需 .actr package 文件即可启动。

风险与处理

风险 影响 处理
Swift async callback 跨 UniFFI 的线程与 Sendable 约束 编译失败或运行时 callback 调度不稳定。 保持 callback interface Send + Sync + 'static，Swift 示例使用 final class 并明确并发安全边界。
没有 package-derived service_spec 服务发现可能只看到 actor type，看不到详细 RPC schema。 第一阶段记录限制；第二阶段提供 explicit service spec API。
错误映射过粗 Swift dispatch 错误可能只表现为 generic internal error。 复用现有 ActrError 桥，并在 route dispatch 示例中明确 unknown route 错误。
API 命名误导为真正 static archive workload 用户可能以为需要 Swift 编译 .a。 文档中统一称为 linked workload 或 static link path，说明它是同进程 callback 模式。

建议里程碑

  1. M1 Rust FFI linked constructor + 编译通过。
  2. M2 Swift high-level API + UniFFI 重新生成。
  3. M3 Swift echo linked workload 示例可运行。
  4. M4 文档补齐 package-backed vs linked/static 的差异。
  5. M5 可选 service_spec 输入，作为后续增强。

这个方案的核心交付是让 Swift 能开发同进程 workload。独立 Swift .a package ABI 不纳入第一阶段。
